Implementation Plan

North Star · v0.1 · 2026-06-25

Context

The repo is greenfield — only docs/PRD.md and docs/mocks/ exist. This plan implements the full product the PRD describes, test-first, across three folders: backend/ (.NET 8), frontend/ (Angular 17.2.2), and e2e/ (Playwright).

Methodology is ATDD: every feature begins with a failing acceptance test, followed by the radically-simplest implementation that makes it pass.

Confirmed decisions

DatabaseSQL Server via EF Core 8.
AI engineStubbed behind interfaces now (seeded fake provider + rules-based matcher). Real Claude API + scraping is a clearly-marked future milestone.
AuthFull ASP.NET Core Identity + JWT + roles (Owner / Child / Mentor) with parent-managed child PIN login.
Run & CIDocker Compose local stack + GitHub Actions (unit / integration / acceptance + Playwright e2e).

Backend — Clean Architecture, file-per-type, SOLID

Dependency rule (inward only): Api → Infrastructure → Application → Domain. Domain depends on nothing.

  • Domain — aggregates & entities: Family, AppUser, ChildProfile, Routine, RoutineLog, Project, Artifact, Activity, ActivityLog, Opportunity, OpportunityMatch, Badge, Milestone, Resource, ConsentRecord. Value objects: Streak, StarPoints, DateRange, GradeBand, MatchScore. One public type per file.
  • Application — vertical slices under Features/<Area>/ with MediatR commands/queries + handlers, DTOs, FluentValidation validators, pipeline behaviors. Defines ports: IFamilyRepository, IChildProfileRepository, IRoutineRepository, IUnitOfWork, ICurrentUser, IClock, IJwtTokenService, IOpportunityProvider, IScholarshipMatcher, IEmailSender.
  • Infrastructure — NorthStarDbContext (SQL Server), one IEntityTypeConfiguration<T> per entity, EF migrations, repositories, ASP.NET Core Identity + role seeding, JWT token service, SeededScholarshipProvider + RulesScholarshipMatcher (AI stubs), NoOpEmailSender.
  • Api — Program.cs composition root, thin controllers (one per aggregate) translating HTTP ↔ MediatR, JWT auth + role policies, global exception middleware, Swagger, health checks, CORS.

ATDD harness

NorthStar.Acceptance.Tests uses xUnit + WebApplicationFactory<Program> driving the real API via HttpClient, against a real SQL Server in Docker through Testcontainers.MsSql. Tests are Given/When/Then with helper builders + FluentAssertions.

Per-feature loop: (1) failing acceptance test → (2) thinnest API→handler→repo→EF slice to go green → (3) refactor + focused unit tests → (4) Angular feature → (5) Playwright e2e for the same flow.

Frontend — Angular 17.2.2, file-per-type, BEM

  • Pinned to 17.2.2, --style=scss --routing. Standalone components, signals, typed reactive forms, lazy-loaded routes.
  • File-per-type / NO single-file components: every component is three files — .ts (always templateUrl/styleUrls), .html, .scss. Enforced via angular.json schematics + an angular-eslint rule.
  • BEM CSS everywhere: block__element--modifier (e.g. .ns-card__header, .task--done). The mock CSS is ported into componentized BEM SCSS + global styles/_tokens.scss & _base.scss.
  • Structure: core/ (auth interceptor, guards, typed API services), shared/ (design-system components ns-card, ns-button, ns-badge, ns-progress, ns-streak, ns-stat, plus sidebar/topbar shells), features/ (auth, dashboard-child, dashboard-parent, routines, portfolio, sports, scholarships, resources, settings), styles/ (tokens, kid vs parent themes).
  • Unit tests: Angular CLI default Karma/Jasmine.

e2e — Playwright, Page Object Model

  • Standalone TypeScript project with @playwright/test + playwright.config.ts (runs against the compose stack).
  • POM: pages/ (LoginPage, ChildDashboardPage, ParentDashboardPage, RoutinesPage, PortfolioPage, ScholarshipsPage, SportsPage, ResourcesPage), components/ (Sidebar, TopBar), fixtures/ (auth + seeded data via API), tests/ (one spec per major flow).
  • Flows: parent register/login → add child → child completes "today's 3 things" & streak increments → create project + artifact/reflection → log sport practice + milestone → scholarship matches save/dismiss + parent approval → browse resources.

Infrastructure — Docker Compose + GitHub Actions

  • docker-compose.yml: mssql (SQL Server 2022, healthcheck), api (multi-stage build, waits for db, applies migrations), web (Angular → nginx), optional mailhog.
  • .github/workflows/ci.yml: jobs — backend (build/test incl. acceptance via Testcontainers), frontend (lint/build/unit), e2e (compose up → Playwright → upload report). On push + PR.

Milestones — each delivered ATDD-style

M0
Walking skeleton & foundations. Solution + 4 projects + test projects, Angular workspace, e2e scaffold, docker-compose, CI skeleton. First red→green: register family owner → login returns JWT + app shell renders. Also renders this HTML plan.
M1
Accounts & family. Parent register/login, create child profile (grade/interests), child PIN login, role enforcement. Auth pages + parent/child shells. e2e: onboarding.
M2
Study routines (core daily loop). Routine templates → "today's plan"; complete routine → stars + streak; weekly completion %. Child Today + Routines screens. e2e.
M3
Projects & portfolio. Create project, add artifact + reflection, timeline, showcase flag. Portfolio screen. e2e.
M4
Sports & activities. Activities/teams, practice/game logs, personal-best milestones, season summary. Sports screen. e2e.
M5
Scholarships (stubbed AI). IOpportunityProvider (seeded) + IScholarshipMatcher (rules: eligibility + horizon). Matched feed, save/dismiss, parent approval, deadline reminders. Finder + approvals screens. Real scraping + Claude extraction deferred behind these interfaces.
M6
Resources, gamification & parent digest. Seeded resources library, badges/milestones, computed weekly digest (email stub). Screens + e2e.
M7
Hardening. COPPA-style consent gating, security pass (rate limiting, validation, RBAC tests), accessibility (WCAG 2.2 AA), observability (Serilog, health), performance.

Verification

  • docker compose up → db + api + web healthy; smoke-check endpoints & UI shell.
  • Backend: dotnet test (unit + acceptance on Testcontainers SQL).
  • Frontend: ng lint, ng test, ng build.
  • e2e: npx playwright test against the compose stack; HTML report.
  • CI green on push/PR (all three jobs). Per milestone: that milestone's acceptance + Playwright flows pass.

Defaults chosen (changeable)

  • Thin controllers over minimal-API endpoint classes.
  • xUnit Given/When/Then over SpecFlow/Reqnroll .feature files.
  • Karma/Jasmine over Jest for Angular unit tests.
  • Testcontainers SQL Server for acceptance tests (vs in-memory/SQLite) for prod fidelity.
